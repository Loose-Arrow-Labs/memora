## Question
Should M10 remain one milestone, or should it split if project import, GitHub
evidence, first-run UI, MCP readiness, and README acquisition work prove too
broad to deliver at first-ten-minute polish?

## Context
The revised M10 scope is strategically correct, but larger than the previous
M10 baseline. It includes new dependency surfaces such as GitHub auth, rate
limits, private repository permissions, import freshness, deleted history, and
partial-import recovery.

## Possible Directions
- keep M10 intact and execute the thinnest real vertical slice across all
  issues
- split after import modes and repository attachment are defined
- move GitHub depth, UI polish, or MCP readiness into a follow-on milestone if
  validation shows the first-run bar cannot be protected otherwise
