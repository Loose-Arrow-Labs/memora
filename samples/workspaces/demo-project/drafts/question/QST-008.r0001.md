## Question

Should Milestone 13 build a custom Memora mobile app as the primary capture
path, or should it instead bridge into existing tools users already have
(Obsidian, Notion, or any markdown folder) via a folder-watch or vault-bridge
ingest path?

## Context

The mobile contribution packet format (docs/mobile-contribution-packet.md) is
already defined as a single markdown file with YAML frontmatter. The transport
is explicitly unspecified in v1.

Users building on AI-assisted workflows typically already have a capture system
in place — Obsidian, Notion, Zettelkasten, or a plain markdown folder. Building
a custom mobile app to produce a markdown file that gets dropped into a folder
duplicates infrastructure those tools already provide, including mobile apps,
sync, and search.

Memora's unique value is governance, lifecycle control, and agent-safe context
retrieval — not capture. Capture is a solved problem for most target users.

The standalone app story makes sense as a full business offering with a complete
workflow, but competes with the growing Obsidian/Notion + MCP ecosystem.

## Possible Directions

- **Vault-bridge / folder-watch (M13 MVP):** watch a designated inbox folder
  (e.g. an Obsidian vault subfolder) for markdown files matching the packet
  format, ingest them automatically as non-canonical planning input. Mobile
  capture = existing tool. No custom mobile app required for v1.
- **Custom mobile app:** build Memora Mobile (Android/iOS) as a first-class
  capture surface. Full control over UX and packet format. Higher build cost
  and competes with mature alternatives.
- **Both, sequenced:** ship vault-bridge first as M13, defer custom mobile app
  to a later milestone once governance and agent workflows are proven.

## Resolution

Open.
