## Context
PR #376 (`feature/get-started-and-installer-package`) introduces the `/get-started`
page, the `LocalOperatorWorkspaceService.CreateProject` method, and the home-page
"Get started" call to action. PR #378 (`feature/promote-proposed-to-draft`)
introduces the proposed-to-draft promotion path that closes the agent loop. PBR-17
extends the `/get-started` flow with GitHub support and runs evidence import end to
end. The handoff names PR #378 as a "honesty" dependency for PBR-17 because the
agent loop must close end to end before PBR-17 can be considered fully done.

Initially the chain for unattended stacked execution was set to
`main → PR #378 → PBR-17`. Reading the actual file diffs showed PR #378's branch
does not contain PR #376's `/get-started` UI surface. PBR-17 builds on top of that
surface, not on top of the approval workflow change.

## Decision
PBR-17 bases on PR #376's branch (`feature/get-started-and-installer-package`) and
targets that branch with its PR. PR #378's contribution is recorded as a
logical-only dependency in PBR-17's PR body, with the honest note that the agent
loop will only close end to end after PR #378 also merges.

This produces the following independent chains under §16 stacked execution:

- `main → PR #376 → PBR-17` (depth two)
- `main → PBR-16` (depth one, independent)
- `main → PBR-06` (depth one, independent; validation may need to wait for the
  others to land before screenshots are honest)

## Alternatives Considered
- Keep PBR-17 on PR #378's branch and duplicate the `/get-started` UI from PR #376.
  Rejected: duplicated UI code is harder to merge cleanly when both land, and the
  duplication has no value as a code dependency.
- Rebase PR #378 onto PR #376 so the chain
  `main → PR #376 → PR #378 → PBR-17` works. Rejected: requires force pushing PR
  #378 and reshaping a peer PR's history; PR #378 is independent and should stay
  that way.
- Create a temporary integration branch merging PR #376 and PR #378, then base
  PBR-17 on it. Rejected: artifact branches that do not correspond to an issue add
  noise to the workflow and complicate cleanup.

## Consequences
- Future stacked decisions must distinguish "code dependency" (target imports or
  calls types from base) from "honest scope dependency" (acknowledged in the PR
  body but no compile dependency). The §16.3 rule already says this; ADR-101
  records a real worked example.
- PBR-17's PR body must call out the PR #378 honest-scope dependency so reviewers
  do not expect PBR-17 alone to close the agent loop.
- This worked example informs the next stacked-mode session: identify the real
  compile dependency before agreeing to a base branch.
