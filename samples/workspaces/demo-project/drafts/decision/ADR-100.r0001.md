## Context
PBR-17 (#372) closes the largest user-felt gap remaining for the Trusted-Circle Beta
milestone: importing a GitHub repository from inside the operator UI without leaving
it. The acceptance criteria require token paste, repository selection, import mode
selection, and an import that produces non-zero evidence on completion. The handoff
in `docs/codex-handoff-trusted-circle-beta.md` is explicit that full OAuth is
post-beta. The existing `GitHubCliEvidenceClient` depends on a locally authenticated
`gh` CLI, which is acceptable for power users but not for the trusted-circle
floor.

## Decision
The trusted-circle GitHub import flow lands with the following shape:

- Authentication is a pasted GitHub personal access token. No OAuth, no browser
  redirect, no Memora-side account. The PAT lives in process memory and on the
  one-shot POST body between step one and step two; it is never written to disk.
- Evidence scope for the new HTTP-based client is issues, pull requests, commits,
  and releases. Reviews, review comments, and discussions remain on the `gh` CLI
  path and are deferred to a follow-up. The `gh` CLI client is unchanged.
- Import execution is synchronous inside the POST handler. After success, the UI
  redirects to `/projects/{id}/first-run-import`, which already shows persisted
  evidence counts and the readiness state. Live rolling counts during import are
  deferred.

## Alternatives Considered
- Full GitHub OAuth in this PR. Rejected: out of trusted-circle scope per handoff;
  doubles the surface area for an unsigned local app.
- Re-skinning the existing `gh` CLI client behind the UI without a new HTTP client.
  Rejected: requires every trusted-circle user to install and authenticate `gh`,
  which the handoff explicitly wants to avoid.
- Full evidence parity with `gh` CLI (reviews, review comments, discussions) in the
  HTTP client. Rejected: too much duplicated parsing for the trusted-circle floor;
  the four endpoints chosen are enough to produce non-zero evidence on completion.
- Asynchronous import with a poll-based progress page. Rejected for this PR:
  acceptable trade-off versus added background-task and status-polling complexity
  for the trusted-circle audience.

## Consequences
- A new `GitHubApiAccountClient` and `GitHubApiEvidenceClient` live alongside the
  existing `GitHubCliEvidenceClient`. The two clients coexist; the UI flow uses the
  HTTP one and the rest of the codebase keeps the CLI one.
- The `/get-started` page presents two real options side by side: local Git path
  and GitHub PAT. The previous "use `gh auth login`" copy is replaced.
- The PR body must acknowledge: synchronous import (no live progress), HTTP client
  scope (issues, PRs, commits, releases only), and the post-beta nature of OAuth
  and organization-level consent.
- Future PRs can extend the HTTP client toward parity with the CLI client without
  changing the UI flow.
